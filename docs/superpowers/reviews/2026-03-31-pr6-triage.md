## What Gets Pulled & How to Adapt

### 1. File Tools → `dagnats-agents/tools/filetools/`

dagnats-agents already has `file_read` as a NATS microservice. Add three more:

**`file_write`** (from PR #6 `agent/tools/file.go` writeFileTool)
- Subject: `tool.exec.file_write`
- Input: `{"path": string, "content": string}`
- Adaptation: Replace in-process `os.WriteFile` with NATS `micro.Handler`
- Add sandbox `ValidatePath` check before any I/O
- Register in `tool_registry` KV with JSON schema

**`file_edit`** (from PR #6 `agent/tools/file.go` editFileTool)
- Subject: `tool.exec.file_edit`
- Input: `{"path": string, "old_string": string, "new_string": string}`
- Adaptation: Same pattern — NATS handler wrapping the exact-string-replace logic
- The uniqueness check (old_string must appear exactly once) is good, keep it
- Add sandbox `ValidatePath` check

**`file_list`** (from PR #6 `agent/tools/file.go` listDirTool)
- Subject: `tool.exec.file_list`
- Input: `{"path": string}`
- Adaptation: NATS handler, bounded at 1000 entries (already in PR #6)
- Add sandbox `ValidatePath` check

### 2. Search Tools → `dagnats-agents/tools/searchtools/` (new package)

**`glob`** (from PR #6 `agent/tools/search.go` globTool)
- Subject: `tool.exec.glob`
- Input: `{"pattern": string, "path": string}`
- Adaptation: NATS handler wrapping `filepath.Walk` + `filepath.Match`
- Keep the bounds: maxResults=500, maxWalk=50000
- Add sandbox `ValidatePath` check on base directory
- Note: PR #6's glob doesn't support recursive `**` patterns (filepath.Match limitation)
  Consider using `doublestar` library or document the limitation

**`grep`** (from PR #6 `agent/tools/search.go` grepTool)
- Subject: `tool.exec.grep`
- Input: `{"pattern": string, "path": string, "file_pattern": string}`
- Adaptation: NATS handler wrapping the file walk + string search
- Keep the bounds: maxResults=200, maxWalk=50000
- Keep the binary file detection heuristic
- Add sandbox `ValidatePath` check
- Note: PR #6 uses `strings.Contains` (literal match only). Consider supporting
  regex via `regexp.MatchString` for more useful search

### 3. Sandbox → `dagnats-agents/tools/sandbox/` (new shared package)

**`ValidatePath`** (from PR #6 `agent/tools/sandbox.go`)
- Shared by filetools, searchtools, and shelltools
- Checks: path within workspace dir OR in allowed paths list
- Rejects path traversal (`../` after Clean)
- Already well-tested in PR #6 (traversal attacks, allowed paths, relative paths)

**`SandboxConfig`** struct:
```go
type SandboxConfig struct {
    WorkspaceDir  string
    AllowedPaths  []string
    BashTimeout   time.Duration
    BashMaxOutput int
}
```

**Critical fix from PR #6 review**: Do NOT use `DefaultSandbox()` with `WorkspaceDir="/"`.
Require explicit workspace dir. Panic or error if empty.

### 4. Existing shelltools Enhancement

dagnats-agents' `shelltools/` already has `shell_exec` with:
- 30s default timeout, 300s max
- 1MB output cap

Pull the sandbox `ValidatePath` check to also validate the working directory.
No other changes needed — shelltools is already more robust than PR #6's bash tool.

## Tool Registration in dagnats-agents

New tools need entries in the `tool_registry` KV bucket. Following the existing pattern:

```json
{
  "name": "file_write",
  "description": "Write content to a file, creating directories as needed",
  "input_schema": {
    "type": "object",
    "properties": {
      "path": {"type": "string", "description": "File path to write to"},
      "content": {"type": "string", "description": "Content to write"}
    },
    "required": ["path", "content"]
  },
  "subject": "tool.exec.file_write"
}
```

Similarly for `file_edit`, `file_list`, `glob`, `grep`.

These should also be added to a `"file-ops-full"` bundle (extending the existing
file-ops bundle) and a new `"search-ops"` bundle.
