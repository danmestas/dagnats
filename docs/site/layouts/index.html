The Problem
Workflow engines got heavy

Every workflow engine today forces you to operate a database cluster. They call it "production-ready" — but what they mean is "ready after you provision Postgres, Redis, Kafka, and a container orchestrator."

  • ×
    Temporal Cassandra or Postgres + Elasticsearch + multi-service cluster. Powerful, but complex to operate.
  • ×
    Hatchet Postgres + RabbitMQ. Closest competitor in the DAG space, but still two databases to keep alive at 3 AM.
  • ×
    Kestra JVM-based. Postgres + Kafka for production. YAML-first approach trades type safety for accessibility.

DagNats takes a fundamentally different approach. NATS JetStream provides streams, KV stores, and consumers — everything a workflow engine needs — in a single embedded dependency.

  • ✓
    Event-sourced by default Every workflow event is immutable. Full replay and audit without a separate system.
  • ✓
    Single binary deployment dagnats serve embeds NATS, engine, API, triggers, and HTTP bridge. One process.
  • ✓
    Zero external dependencies No web framework, no ORM, no logging library. Pure Go + NATS. Nothing else.

Capabilities
Primitives, not plugins
↺
Agent Loops
First-class iterative execution with Continue(), max iterations, checkpoints, and duration limits. Built for LLM reasoning chains.
◆
Dynamic DAGs
Planner steps output JSON DAG fragments at runtime. Engine materializes them as real steps. Workflows that design themselves.
⊚
Singleton Execution
At most one active run per key. Skip duplicates or cancel-and-replace. KV-based locks with stale detection.
◷
Durable Sleep
Sleep up to 365 days via NakWithDelay. No background goroutines, no timer leaks, no resource consumption while waiting.
⇄
Sticky Workers
Soft or hard worker affinity. Route subsequent steps to the same worker for cache locality or stateful agent sessions.
✓
Human Approvals
Pause for human review. 256-bit tokens, configurable timeouts up to 7 days, auto-reject on expiry. Slack/Discord hooks built in.
↶
Saga Compensation
Automatic rollback in reverse topological order. Declarative compensate steps. The engine handles sequencing.
◈
Event Correlation
WaitForEvent with field matching. CancelOn for reactive termination. O(1) correlator lookup per event type.
☷
Flow Control
Three-scope concurrency limits, priority queues with numeric offsets, token bucket rate limiting, and idempotency keys.
Developer Experience
Define workflows in Go
workflow.go 
wf := dag.NewWorkflow("deploy-pipeline").
    WithTimeout(30 * time.Minute).
    WithSingleton(dag.SingletonModeCancel).
    WithConcurrency(5, 0).
    WithPriority(dag.PriorityConfig{
        Key:   "data.tier",
        Rules: map[string]int{
            "enterprise": 300,
            "pro":        60,
        },
    })

wf.Step("validate", "validate-input")
wf.Step("build", "docker-build").
    DependsOn("validate")
wf.Step("approve", "approval").
    DependsOn("build").
    AsApproval(24 * time.Hour)
wf.Step("deploy", "k8s-deploy").
    DependsOn("approve").
    WithCompensate("rollback")
worker.go 
w := worker.New(nc, "deploy-worker")

worker.HandleTyped(w, "docker-build",
    func(ctx worker.TaskContext,
         in BuildInput) (BuildOutput, error) {

        // Checkpoint survives crashes
        ctx.Checkpoint(map[string]any{
            "stage": "building",
        })

        // Stream logs in real-time
        ctx.PutStream([]byte("Building image..."))

        img, err := buildImage(ctx, in.Repo)
        if err != nil {
            return BuildOutput{}, err
        }

        return BuildOutput{Image: img}, nil
    },
)
