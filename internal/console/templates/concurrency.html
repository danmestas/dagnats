Blocked runs

The engine does not emit which gate a given run is waiting on — neither the admission-gate KV nor a run's state carries the run-to-gate blocking relationship, and there is no "blocked" run or step status to join on. Rather than fabricate that linkage, this section is intentionally omitted until the engine emits a wait-queue.

Every admission gate above is lazy — singleton locks, task-concurrency counters, rate-limit token buckets (rate_limits), and trigger-debounce windows (debounce_state) are empty in KV until a workflow hits contention.
